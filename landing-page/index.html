Git-driven workflow

Everything you need for Target GitOps

A complete pipeline from activity scaffolding to production deploy — with validation gates, duplicate protection, and traceable naming.

📁

Activity-as-Code

Each Target activity is a self-contained folder with activity-info.json and HTML offer files. No central registry required.

🔄

Create & Update

Deploy brand-new activities on first merge, then update existing HTML offers when activity_id and offer_id are saved.

🔍

Auto-Discovery

Activity folders are discovered automatically from the repository. Global settings live in deploy.config.json only.

✅

CI Gate

Pull requests fail unless they include a valid activity create or update. Prevents empty merges that would skip deploy.

🛡️

HTML Validation

HTML5 structure, embedded CSS and JavaScript syntax checks, plus Prettier formatting — all enforced before merge.

🔐

Secure Auth

GitHub Secrets for Adobe IMS credentials with automatic token refresh on 401 using client credentials.

Pros & Cons

An honest look at deploying Adobe Target activities through GitHub and MCP — built for time savings and team efficiency.

Pros

Why teams choose this approach

  • ✓
    Version-controlled activities Every HTML offer and config change is tracked in Git with full history and blame.
  • ✓
    Automated deploy on merge No manual copy-paste into Adobe Target — merge to main and the pipeline runs.
  • ✓
    Major time savings Time Saver Skip repetitive Target UI steps — no manual offer creation, HTML upload, or variant attachment on every change.
  • ✓
    Faster go-live cycles Efficiency Edit HTML locally, open a PR, and merge. Deploy completes in minutes instead of a long manual Target workflow.
  • ✓
    Efficient team collaboration Efficiency Multiple marketers and developers work in parallel on branches — reviews happen in GitHub, not over email or screenshots.
  • ✓
    Team review before deploy Quality Gate Your team reviews your code in pull requests first. Only after approval and merge to main is the activity pushed to Adobe Target.
  • ✓
    Pull request validation Broken HTML, invalid JSON, or empty merges are caught before they reach production.
  • ✓
    Official Adobe Target MCP Uses the native MCP server — no custom APIs or brittle screen-scraping integrations.
  • ✓
    Zero per-activity registration Add a folder with activity-info.json and it is auto-discovered. No central config edits.
  • ✓
    Traceable naming Activities and offers are prefixed with [GitHub][username] for clear audit trails.
  • ✓
    Create and update in one workflow First deploy creates the activity; later commits update offers without rebuilding from scratch.
  • ✓
    One-command activity setup Time Saver Scaffold a full activity folder in seconds with create_activity.py — no boilerplate typing or config hunting.
  • ✓
    Safe by default Activities are not auto-activated — content lands in a saved state for review in the Target UI.
Cons

Things to be aware of

  • !
    Requires saved IDs for updates After the first deploy, you must copy activity_id and offer_id back into the repo.
  • !
    Activity changes only Docs-only or script-only merges fail CI — every merge to main needs an activity change.
  • !
    Adobe credentials setup GitHub Secrets and IMS client credentials must be configured before the first deploy works.

Quick Start

Go from zero to a deployed Adobe Target activity in five steps.

1

Clone & configure

Clone the repository and set your deploy username in deploy.config.json.

git clone https://github.com/adobemohit/analogDevices.git cd analogDevices
2

Add GitHub Secrets

In Settings → Secrets and variables → Actions, add your Adobe IMS credentials.

Secret Description
ADOBE_ACCESS_TOKEN Adobe IMS bearer token for Target MCP
ADOBE_CLIENT_ID Adobe Developer Console client ID
ADOBE_CLIENT_SECRET Adobe Developer Console client secret
3

Create your first activity

Scaffold a new activity folder from the built-in template.

python scripts/create_activity.py summer_promo_xt_test "Summer Promo XT Test"
4

Open a pull request

Edit the HTML offer, commit, and open a PR. CI validates activity changes and code quality before merge.

5

Merge & save IDs

After merge to main, the deploy job publishes to Adobe Target. Copy activity_id and offer_id from the Actions log into activity-info.json for future updates.

{ "activity_id": 123456, "variants": [{ "offer_id": 789012 }] }
